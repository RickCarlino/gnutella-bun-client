# WebRTC Transfer Extension

This document proposes a Gnutella extension for transferring a selected `QueryHit` result over WebRTC.

It leaves query routing, result routing, and the file transfer format intact. The new pieces are:

- `RTCQ` in `Query`
- `RTCH` in `QueryHit`
- a small HTTP signaling API
- one ordered WebRTC data channel carrying the usual download HTTP bytes

Signaling is mediated by a third-party rendezvous endpoint that both peers can reach outbound. That keeps the setup path workable even when both peers are firewalled for inbound TCP.

In scope:

- keep resource discovery in `Query` and `QueryHit`
- move offer and answer exchange onto HTTP rendezvous endpoints
- carry the file transfer itself over one ordered WebRTC data channel

The rendezvous server carries signaling only. File bytes move directly between downloader and uploader.

## Why

The classic transfer path is direct TCP HTTP, with `PUSH` as the fallback when the uploader cannot accept an inbound connection. That works well when at least one side is reachable. It is much less reliable when both sides sit behind home NAT or other inbound filtering.

This proposal adds a second transport:

- the search path stays Gnutella
- the signaling path is HTTP
- the transfer path is peer-to-peer WebRTC

The bytes carried over the data channel are the same HTTP request and response bytes already used for `/get`.

## Overview

The protocol involves three actors:

- `downloader`: the node that issued the query and chose the result
- `uploader`: the node that returned the chosen result
- `rendezvous endpoint`: a third-party HTTP endpoint used to pass the offer and answer between them

The flow is:

1. the downloader sends a `Query` containing `RTCQ`
2. the uploader returns a `QueryHit` containing `RTCH`
3. the downloader chooses one rendezvous endpoint advertised in that `RTCH`
4. the downloader creates an offer and posts it to `/rtc/offer`
5. the uploader receives or polls that offer, creates an answer, and posts it to `/rtc/answer`
6. the downloader applies the answer and opens the data channel
7. the downloader sends the normal download request over that channel
8. the uploader sends the normal HTTP response over that channel

## Advertisement

### `RTCQ` in `Query`

The downloader adds one GGEP block named `RTCQ` to the outgoing `Query`.

`RTCQ` is a capability marker. It tells responders that the query origin can continue a selected result over the WebRTC upgrade path if the hit advertises `RTCH`.

`RTCQ` carries no rendezvous data. Its job is only to make the downloader's support explicit on the query that produced the later hit.

Current TLVs:

| TLV | Hex  | Meaning                                  |
| --- | ---- | ---------------------------------------- |
| VER | `01` | Extension version. Current value is `1`. |

### `RTCH` in `QueryHit`

The uploader adds one GGEP block named `RTCH` to the private area of the `QueryHit`.

`RTCH` makes a specific result RTC-eligible. It says three things about that hit:

- the uploader is prepared to accept an RTC offer for this result
- the uploader has included the cookie needed to validate that offer
- the hit names one or more rendezvous endpoints that can carry the offer and answer

The eligibility rule is:

- a downloader sends an RTC offer only when the query carried `RTCQ`
- a hit is RTC-eligible only when it carries `RTCH` and at least one `RTEP`

Current TLVs:

| TLV    | Hex  | Meaning                                                  |
| ------ | ---- | -------------------------------------------------------- |
| VER    | `01` | Extension version. Current value is `1`.                 |
| COOKIE | `05` | Short-lived query-bound token generated by the uploader. |
| RTEP   | `0f` | Repeated rendezvous endpoint records.                    |

Fixed data-channel profile:

- `label = "GNUT"`
- `protocol = ""`
- ordered delivery

## Rendezvous Endpoints

The extension advertises concrete rendezvous sockets. An `RTEP` is an IPv4 socket where the HTTP signaling API is reachable and where the uploader can poll for pending offers.

Each `RTEP` payload is exactly 7 bytes:

```text
reserved(1) | ipv4(4) | port(2)
```

Rules:

- `reserved` is `0x00` in version 1
- `ipv4` is the rendezvous server IPv4 address in network byte order
- `port` is the rendezvous server TCP port in network byte order
- entries with the wrong length are invalid
- entries with a non-zero reserved byte should be ignored for now

Example:

```text
00 cb 00 71 07 18 ca
```

This means:

- reserved byte `00`
- IPv4 `203.0.113.7`
- port `6346`

`RTCH` should carry at least one `RTEP`.

If a node cannot advertise at least one reachable rendezvous endpoint, it should not advertise `RTCH` for this extension.

For version 1, transmitted order is preference order. The downloader should try `RTEP` entries in the order advertised.

## Cookie

`RTCH` carries a short-lived `COOKIE` generated by the uploader. The downloader echoes it back in the later offer.

In `gnutella-bun`, the cookie is 20 bytes:

- 4-byte big-endian expiry timestamp in seconds
- 16-byte HMAC tag

The HMAC input is:

```text
"rtc-upgrade/1-cookie" || query_descriptor_id || expiry
```

This is a lightweight anti-stale-offer token. It lets the uploader reject random or expired offers without keeping per-hit state.

## Signaling Transport

The offer and answer move over HTTP. All endpoints are rooted at the chosen rendezvous host and port.

There is no JSON signaling layer in this draft. The body is SDP. HTTP only carries the minimal routing and correlation metadata needed around that SDP.

Endpoint paths:

- `POST /rtc/offer`
- `GET /rtc/offer?target=<servent_id_hex>`
- `POST /rtc/answer`
- `GET /rtc/answer?rid=<rid_hex>&token=<token_hex>`

### `POST /rtc/offer`

The downloader posts the raw offer SDP as the request body.

Example:

```http
POST /rtc/offer HTTP/1.1
Content-Type: application/sdp
X-RTC-Target: 9999aaaabbbbccccddddeeeeffff0000
X-RTC-File-Index: 7
X-RTC-Query-ID: 8899aabbccddeeff0011223344556677
X-RTC-RID: 11112222333344445555666677778888
X-RTC-Cookie: 001122...
X-RTC-Token: 1234567890abcdef1234567890abcdef

v=0
...
```

The body is the full SDP offer. The metadata moves in request headers:

| Header             | Meaning                                                                                              |
| ------------------ | ---------------------------------------------------------------------------------------------------- |
| `X-RTC-Target`     | Uploader servent ID. 16 bytes as 32 hex characters.                                                  |
| `X-RTC-File-Index` | Selected result index on the uploader.                                                               |
| `X-RTC-Query-ID`   | Original query descriptor ID. 16 bytes as 32 hex characters.                                         |
| `X-RTC-RID`        | Per-attempt RTC request ID. 16 bytes as 32 hex characters.                                           |
| `X-RTC-Cookie`     | Echo of the uploader's `RTCH` cookie. 20 bytes as 40 hex characters.                                 |
| `X-RTC-Token`      | Session token used to authorize answer delivery and answer retrieval. 16 bytes as 32 hex characters. |

Response codes:

- `202 Accepted` when the offer is queued
- `400 Bad Request` when the headers or SDP body are invalid

### `GET /rtc/offer?target=<servent_id_hex>`

The uploader polls for queued offers addressed to that servent ID.

On `200 OK`, the response body is the raw SDP offer and the metadata appears in response headers. The uploader uses that metadata to identify the selected file, validate the cookie against the original query, and associate the later answer with the same `rid`:

```http
HTTP/1.1 200 OK
Content-Type: application/sdp
X-RTC-File-Index: 7
X-RTC-Query-ID: 8899aabbccddeeff0011223344556677
X-RTC-RID: 11112222333344445555666677778888
X-RTC-Cookie: 001122...
X-RTC-Token: 1234567890abcdef1234567890abcdef

v=0
...
```

Response codes:

- `200 OK` with one queued offer
- `204 No Content` when no offer is waiting
- `400 Bad Request` when the query parameter is malformed

### `POST /rtc/answer`

The uploader posts the raw answer SDP as the request body.

Example:

```http
POST /rtc/answer?rid=11112222333344445555666677778888&token=1234567890abcdef1234567890abcdef HTTP/1.1
Content-Type: application/sdp

v=0
...
```

Response codes:

- `202 Accepted` when the answer is stored
- `404 Not Found` when the session token is unknown
- `400 Bad Request` when the query parameters or SDP body are invalid

### `GET /rtc/answer?rid=<rid_hex>&token=<token_hex>`

The downloader polls for the answer associated with its earlier offer.

On `200 OK`, the response body is the raw SDP answer with `Content-Type: application/sdp`.

Response codes:

- `200 OK` with the answer
- `204 No Content` when the answer is not ready yet
- `404 Not Found` when the session is unknown
- `400 Bad Request` when the query parameters are malformed

## Downloader Behavior

After selecting an RTC-capable result, the downloader:

1. chooses a rendezvous endpoint
2. creates an `RTCPeerConnection`
3. creates one ordered data channel named `GNUT`
4. creates an offer
5. posts that offer SDP to `/rtc/offer`
6. polls `/rtc/answer`
7. applies the answer
8. waits for the data channel to open
9. sends the normal HTTP download request over that channel

## Uploader Behavior

The uploader:

1. polls each rendezvous endpoint it advertises
2. receives one offer request
3. validates the echoed cookie against `queryIdHex`
4. creates an `RTCPeerConnection`
5. applies the offer and creates an answer
6. posts the answer SDP to `/rtc/answer`
7. waits for the incoming data channel
8. reads one HTTP request from the channel
9. serves the selected file over the same channel

## Data Channel Profile

The data channel carries one HTTP transaction as a byte stream:

- the label is always `GNUT`
- the protocol string is always empty
- delivery is ordered
- the downloader sends one `GET /get/...` request
- the uploader sends one HTTP status line, headers, and response body

This preserves existing download semantics:

- range requests still work
- resume still works
- the file parser and response parser do not need a second transfer format

The current implementation chunks outbound data-channel messages at 16 KiB and reconstructs the HTTP byte stream on receipt.

## STUN

Each side uses its own local STUN configuration.

STUN servers are not signaled in-band.

For version 1:

- only `stun:` URLs are relevant
- TURN is out of scope

## Recommended Limits

The wire format is intentionally small. A reasonable version 1 profile is:

- at most 4 explicit `RTEP` records per hit
- short HTTP timeouts for rendezvous requests
- no redirect following on rendezvous HTTP requests

## End-to-End Example

Peer `A` queries. Peer `B` returns a hit. `B` is RTC-capable and advertises one rendezvous endpoint:

- `198.51.100.20:8080`

The flow is:

1. `A` sends `Query + RTCQ`
2. `B` sends `QueryHit + RTCH + RTEP(198.51.100.20:8080)`
3. `A` chooses `198.51.100.20:8080`
4. `A` posts `/rtc/offer`
5. `B` polls or receives that offer, validates the cookie, and posts `/rtc/answer`
6. `A` polls `/rtc/answer`, applies the answer, and opens the data channel
7. `A` sends the normal download request over the data channel
8. `B` sends the normal HTTP response over the data channel
